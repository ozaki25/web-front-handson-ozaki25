# lesson98: セマンティック HTML とアクセシビリティの基礎

## ゴール

- なぜアクセシビリティ（a11y）に配慮するかを自分の言葉で説明できる
- WCAG とは何か、準拠レベル（A / AA / AAA）の違いを説明できる
- セマンティック HTML が a11y の土台になる理由を理解する
- コントラスト比の基準（AA / AAA）を知り、DevTools / Lighthouse で確認できる

## 解説

### なぜアクセシビリティ（a11y）に配慮するか

アクセシビリティ（accessibility、略して **a11y**）は「**すべてのユーザーがウェブを使える状態** にする」取り組みです。対象は次のような利用者です。

- **視覚に関するもの**: 全盲 / 弱視 / 色覚特性のある人（スクリーンリーダーを使う、拡大表示する、コントラストが低いと読めない）
- **聴覚に関するもの**: 聴覚を使いにくい人（動画の字幕が必要）
- **運動に関するもの**: マウスを使いにくい人（キーボードや音声コマンドで操作する）
- **認知に関するもの**: 複雑な UI や素早い動きが苦手な人

さらに **誰にとっても役立つ** 場面も多くあります。

- スマホを片手で操作する状況（大きなタップ領域が欲しい）
- 日差しの強い屋外（高コントラストが欲しい）
- 満員電車で音が出せない（字幕が欲しい）
- キーボード派の開発者（Tab で操作したい）

### WCAG とは何か

**WCAG**（Web Content Accessibility Guidelines）は、W3C が策定する Web アクセシビリティの国際標準です。「何をどの程度すれば、誰でも使えるページと言えるか」を具体的な達成基準として定義しています。

2026 年現在の最新版は **WCAG 2.2**（2023 年勧告）で、前バージョン 2.1 から 9 件の達成基準が追加されました。

#### 準拠レベル: A / AA / AAA

WCAG の達成基準は 3 段階のレベルに分類されています。

| レベル | 位置づけ | 法的・実務の文脈 |
|---|---|---|
| **A**（最低限） | 障壁が大きすぎて一部の人がまったく使えない状態を排除する | これを満たさないと法令上「問題あり」とされる国も多い |
| **AA**（実用水準） | ほとんどの利用者に対応できる水準 | 日本の「令和元年改正 JIS X 8341-3」や EU の欧州アクセシビリティ法はこの水準を要求。政府・公共機関サイトや大企業サイトでは事実上の必須 |
| **AAA**（高水準） | さらに細かいニーズまでカバー | 全コンテンツへの適用は難しい場合も多く、「できる範囲で対応」を推奨する位置づけ |

「AA 準拠」というのが現代の実務の基準線です。

#### WCAG の 4 原則（POUR）

WCAG の達成基準はすべて次の 4 原則のどれかに属します。

| 原則 | 意味 | 代表的な基準 |
|---|---|---|
| **Perceivable**（知覚可能） | 情報を何らかの方法で受け取れる | `alt` テキスト / キャプション / コントラスト比 |
| **Operable**（操作可能） | UI を操作できる | キーボード操作 / フォーカス管理 / タイムアウトの配慮 |
| **Understandable**（理解可能） | 情報と操作が理解できる | ラベル / エラーメッセージ / 言語指定（`lang`） |
| **Robust**（堅牢） | 幅広い環境で解釈できる | 有効な HTML / ARIA の正しい利用 |

これらは「ルールを守る」というより「ユーザーが何で詰まるかを 4 軸で考える」ための枠組みです。

### セマンティック HTML が土台

「HTML / CSS 基礎」のレッスン群（lesson04 〜 lesson06）でランドマーク要素・`alt` 属性・`<label>` を学びましたが、それらが a11y の観点でも「正解」である理由は WCAG の達成基準に直結しているからです。

| 章 1 で学んだこと | 対応する WCAG 基準 |
|---|---|
| `<nav>` / `<main>` / `<header>` でページを分ける | 4.1.2 名前・役割・値（Robust） |
| `<h1>` → `<h2>` → `<h3>` の順で見出しを使う | 2.4.6 見出しおよびラベル（Understandable） |
| `alt` で画像の意味を伝える | 1.1.1 テキストによる代替（Perceivable） |
| `<label>` でフォームと入力欄を紐付ける | 1.3.1 情報および関係性（Perceivable） |
| `<html lang="ja">` で言語を明示する | 3.1.1 ページの言語（Understandable） |

**「a11y のために何か特別なことをする」ではなく、「正しい HTML を書く」ことが大半の a11y 要件を満たす** というのが重要な認識です。

### コントラスト比: WCAG が数値で定める基準

コントラスト比は、WCAG の中でも数値で検証できる数少ない基準のひとつです。文字色と背景色の明るさの差を比率で表し、読みやすさの最低基準を定めています。

| 対象 | AA（実用レベル） | AAA（高水準） |
|---|---|---|
| 通常のテキスト（18pt 未満 / 14pt 未満の太字） | **4.5:1** 以上 | 7:1 以上 |
| 大きなテキスト（18pt 以上 / 14pt 以上の太字） | **3:1** 以上 | 4.5:1 以上 |
| UI 部品の境界線・アイコン | **3:1** 以上 | — |

よくある事故として、`#808080`（グレー）を白背景（`#ffffff`）に使うとコントラスト比は約 3.95:1 で、通常テキストの AA（4.5:1）を満たしません。ダークモードで同じ色を使うと黒背景（`#1a1a1a`）でも不足することがあります。

### DevTools / Lighthouse でコントラスト比を確認する

コントラスト比は目視では判断できません。ツールを使って測ります。

**Chrome DevTools の場合:**

1. Elements タブで文字要素を選ぶ
2. Styles ペインの `color` プロパティをクリックしてカラーピッカーを開く
3. ピッカー下部に `Contrast ratio: 5.74` のように自動で表示される
4. AA / AAA のアイコン（チェックマーク）で合否も分かる

**Lighthouse の場合:**

1. DevTools → Lighthouse タブ
2. 「Accessibility」だけ選択して「Analyze page load」
3. 「Contrast」カテゴリに違反がある要素が一覧で出る

**その他のツール:**

- [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/) — 色コードを入れるだけで即計算
- Figma のコントラストチェッカープラグイン — デザイン段階で確認できる

### 自動チェックの限界

Lighthouse や axe などの自動チェックツールが検出できるのは、WCAG の問題のうち約 **57%** と axe-core 公式ドキュメントは説明しています（残りは人間の判断が必要）。コントラスト比・`alt` の欠落・`<label>` の欠落は自動検出できますが、「`alt` の内容が適切か」「キーボード操作の順序が自然か」「エラーメッセージが分かりやすいか」は人間が確認するしかありません。

**自動チェックはゼロの確認、人間のレビューは質の確認** と役割を分けて運用するのが実際的です。

## 演習

### ゴール

- 実際のサイトで Lighthouse の Accessibility スコアを確認する
- コントラスト比の違反がある要素を DevTools で見つける

### 手順 1: Lighthouse で Accessibility スコアを測る

1. 本教材サイト（または自分が作ったページ）を開きます
2. DevTools → Lighthouse タブを開きます
3. カテゴリは「Accessibility」だけチェックを残し、他は外します
4. 「Analyze page load」を押します
5. スコアと「Failing Elements」の一覧を読みます

### 期待出力

- スコアが数値で表示される（0〜100）
- 「Contrast」セクションに違反があれば、要素が色付きでハイライトされる
- 問題がなければ「Passed Audits」に移動していることを確認できる

### 変える

CSS の `color` を `#808080` に変えて再測定します。コントラスト比が 3.95:1 になり、Lighthouse が「Background and foreground colors do not have a sufficient contrast ratio」と警告することを確認します。

### 自分で書く

- よく使うサービス（GitHub / Twitter / Notion など）を Lighthouse で測り、スコアを比較する
- DevTools の Elements タブで、コントラスト比が AA 未満の要素を 1 つ見つけて、AA を満たす代替色を提案する

## まとめ

- アクセシビリティは「誰でも使える」状態を目指すこと。対象は視覚・聴覚・運動・認知のあらゆる利用者
- **WCAG 2.2** が国際標準。**AA 準拠**が政府・大企業サイトの実務の基準線
- 達成基準は **POUR（知覚可能・操作可能・理解可能・堅牢）** の 4 原則に分類される
- 「正しいセマンティック HTML を書く」ことが WCAG 達成基準の大半を満たす近道
- コントラスト比は通常テキストで **4.5:1**（AA）以上が必須。DevTools / Lighthouse / WebAIM で測れる
- 自動ツールは WCAG 問題の約 57%（axe-core 公称）しか検出できない。人間によるレビューと組み合わせる
