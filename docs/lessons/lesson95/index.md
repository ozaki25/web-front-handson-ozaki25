# lesson95: CDN の仕組みと Vercel Edge Network

## ゴール

- CDN がなぜページを速く返せるのかを説明できる
- `Cache-Control` の `s-maxage` と `private` が CDN 文脈でどう効くかを理解する
- CDN キャッシュのパージと「デプロイしたのに反映されない」を結びつけられる
- Vercel にデプロイしたページで `x-vercel-cache` / `Age` ヘッダを読める

## 解説

「HTTP キャッシュ」のレッスンで学んだ `s-maxage` と `private` は、CDN を想定した設定です。本レッスンではその CDN 自体の仕組みを押さえます。

### CDN とは何か

CDN（Content Delivery Network）は、世界中に分散して置かれた **エッジサーバー** のネットワークです。ユーザーのリクエストは、遠い日本のオリジンサーバーではなく、**地理的に近い**エッジサーバーから返されます。

```
ユーザー（東京）
  ↓ リクエスト
エッジサーバー（東京）← キャッシュがあればここで返す
  ↓ キャッシュなし・期限切れ時のみ
オリジンサーバー（米国・東海岸など）
```

速い理由は 2 つです。

1. **物理的な距離が短い**: パケットの往復時間（RTT）は距離に比例します。東京↔米国は約 150ms かかりますが、東京↔東京なら数 ms です
2. **キャッシュヒット時はオリジンに届かない**: 静的アセット（JS / CSS / 画像）は一度エッジに乗ると、以降はオリジンに問い合わせないためオリジンの処理時間も節約できます

### `s-maxage` と `private` の意味

`Cache-Control` には「ブラウザ向け」と「CDN 向け」の両方を制御するディレクティブがあります。

| ディレクティブ | ブラウザ | CDN / 中間キャッシュ |
|---|---|---|
| `max-age=N` | N 秒キャッシュ | N 秒キャッシュ（`s-maxage` がなければ） |
| `s-maxage=N` | 無視 | N 秒キャッシュ |
| `private` | キャッシュ可 | **キャッシュ禁止** |
| `public` | キャッシュ可 | キャッシュ可 |

**`private`** は「このレスポンスは特定のユーザーに紐付いている（ログイン状態など）のでCDN に持たせてはいけない」という意味です。CDN が `private` なレスポンスをキャッシュすると、別のユーザーに他人のデータを返してしまうため、必ず付けます。

**`s-maxage`** は「CDN にはこの秒数キャッシュしてよいが、ブラウザ側の挙動は `max-age` で別に制御したい」という場合に使います。例えば次の設定は、ブラウザは 60 秒キャッシュし、CDN は 1 日キャッシュします。

```
Cache-Control: public, max-age=60, s-maxage=86400
```

### CDN キャッシュのパージ

CDN はキャッシュを**エッジサーバー上に保持する**ため、オリジンのコンテンツを更新しても CDN が古いキャッシュを返し続けることがあります。これは「デプロイしたのに反映されない」の CDN 版として現れます。

```
ユーザー → エッジ（古いキャッシュ）→ 古い内容が返る
                ↑
        オリジンは更新済みだが、エッジに届いていない
```

解決策は 2 つです。

**1. デプロイ時に自動パージ（Vercel の場合）**

Vercel は新しいデプロイのたびに CDN キャッシュを自動でパージします。ファイル名にハッシュが入った静的アセット（`/assets/main.8c29f6e4.js` など）は URL 自体が変わるため、そもそもパージ不要です。

**2. キャッシュ有効期限を短くする**

`s-maxage=3600` のように短い期限にしておくと、古いキャッシュが残る時間を抑えられます。

### Vercel Edge Network

Vercel は世界 100 箇所以上にエッジサーバーを持ちます。ページを開くと、最も近いエッジが応答します。本コースの教材サイトも Vercel にデプロイしているため、日本のユーザーには東京リージョンのエッジから返っています。

Vercel は静的ファイル（VitePress のビルド出力）に対して、`Cache-Control` と `x-vercel-cache` が付いたレスポンスを返します。`Cache-Control` の具体的な値はフレームワークやファイル種別で変わります（例: ハッシュ付きアセットは `public, max-age=31536000, immutable` のような長期キャッシュ、HTML は短め）。

```
Cache-Control: <Vercel が決める値>
x-vercel-cache: HIT
```

ユーザーが細かく設定しなくても、CDN とブラウザの両方でうまくキャッシュが効くようになっています。

### レスポンスヘッダで CDN を読む

DevTools の Network タブでリソースを選ぶと、CDN が付けたヘッダが確認できます。

| ヘッダ | 意味 |
|---|---|
| `x-vercel-cache` | `HIT`（エッジのキャッシュから返した）/ `MISS`（オリジンに取りに行った）/ `STALE`（古いキャッシュを返しつつ裏で再検証）/ `PRERENDER`（ビルド時に生成された静的版） |
| `Age` | このレスポンスが CDN にキャッシュされてから何秒経過したか |
| `x-cache` | Vercel 以外の CDN（CloudFront など）でも同様のヘッダが使われる |

`Age: 3821` であれば「このレスポンスは約 64 分前にエッジに乗った」という意味です。`Age` が大きいほど、古いキャッシュが返ってきている可能性を示します。

## 演習

### ゴール

- Vercel にデプロイした本教材サイトで `x-vercel-cache` / `Age` ヘッダを DevTools から読む
- `HIT` と `MISS` の両方を観察する

### 手順 1: DevTools で観察する

1. 本教材サイト（Vercel にデプロイされた URL）を開きます
2. F12（または `Cmd+Opt+I`）→ Network タブを開きます
3. ページをリロードします
4. 一覧から HTML ファイル（一番上の行）をクリックします
5. 右パネルの `Headers` → `Response Headers` を確認します

次を見てください。

- `x-vercel-cache` の値（`HIT` / `MISS` / `STALE` / `PRERENDER`）
- `Age` の値（`HIT` の場合に数値が入っている）

### 期待出力

- `x-vercel-cache: HIT` の場合: `Age: ○○`（秒数）が表示される
- `x-vercel-cache: MISS` の場合: `Age: 0` またはヘッダなし
- 静的アセット（`/assets/...js` など）も同様に `HIT` になることを確認する

### 変える

`Hard reload`（`Ctrl+Shift+R` / `Cmd+Shift+R`）でリロードします。`x-vercel-cache` が `MISS` に変わることがあります（ブラウザのキャッシュを使わない強制リクエストになるため）。

通常リロードに戻して数秒おいてから確認すると、`HIT` に戻ることが多いです。

### 自分で書く

- `private` と `public` の違いを、「ログイン後のマイページ」と「トップページ」を例にして 2〜3 文で説明してみる
- `Age: 7200` が返ってきたとき、このリソースが CDN にキャッシュされたのはいつごろか計算する（現在時刻 − 7200 秒）

## まとめ

- CDN はエッジサーバーが地理的に近い場所でレスポンスを返すことで速度を上げる
- `s-maxage` は CDN 向けのキャッシュ期限。ブラウザの `max-age` とは別に指定できる
- `private` は CDN へのキャッシュを禁止する。ユーザー固有のコンテンツに必ず付ける
- CDN キャッシュが古い場合は **パージ** が必要。Vercel はデプロイ時に自動パージする
- `x-vercel-cache: HIT` / `Age` ヘッダで CDN キャッシュの状態を確認できる
