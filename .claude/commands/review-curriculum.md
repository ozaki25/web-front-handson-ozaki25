# カリキュラムレビュー（4 視点並列）

引数: `$ARGUMENTS`（例: `docs/lessons/lesson47/index.md` または `4 章`）

対象を 4 エージェント並列で辛口レビューし、指摘を統合してレポートする。過去のサイクルで実際に回った 3 視点に加え、図表・デモ・例の伝達力を見る「通読 UX」視点を 4 つ目として固定化したもの。

## 起動手順

1. レビュー対象のファイル / 範囲を `$ARGUMENTS` から決める（ファイル、ディレクトリ、章番号のいずれでも対応）
2. 以下 4 つのエージェントを **`run_in_background: true` で並列起動** する（`subagent_type: general-purpose`）
3. 4 エージェントすべて完了したら、収束した指摘を **Critical / High / Middle** で統合し、Top 5 を抽出してユーザーに報告

```

## 必読ファイル
1. /home/user/web-front-handson-ozaki25/CLAUDE.md
2. レビュー対象: <対象ファイル>
3. 前後のレッスン（該当する場合）

## 役割
想定読者は「HTML / CSS / JS / React / Next.js をほぼ触ったことがない独学者」。完走可能性を最優先に評価。

## 観点
1. 脱落ポイント（レッスン番号 + 理由）
2. 前提の飛躍（前レッスンで未説明の概念を突然使っているか）
3. 期待出力の明確さ
4. コードの省略 `…(省略)…` の有無
5. 螺旋構造の機能（前後の接続）
6. 統合レッスンの敷居（前章までの知識で解けるか）
7. モチベーション維持
8. トーンの一貫性

## 出力（2000〜4000 字）
- 総評（300〜500 字）
- Critical（3〜7 件）: lessonXX の引用 + 改善案
- High（5〜12 件）: 同
- Middle（3〜6 件）
- 総合評価: 完走可能性（10 段階）/ 公開可能レベル判定 / 次サイクルで必ず潰すべき Top 5

辛口、具体的、絵文字禁止。
```

## エージェント 2: 技術正確性・最新 API 準拠視点

```
あなたは Web フロントエンド学習コンテンツの **レビュアー（技術正確性、React 19 / Next.js 15 準拠）** です。コード変更は行わず、読んでレビューするだけ。

## 必読ファイル
1. /home/user/web-front-handson-ozaki25/CLAUDE.md
2. レビュー対象: <対象ファイル>

## 重点チェック
- React 19: `useActionState`（`react` から、シグネチャ `[state, formAction, isPending] = useActionState(action, initialState)`）、`useFormStatus`（`react-dom` から）、`ref` の props 化（`forwardRef` 不要）、React Compiler への言及
- Next.js 15: `params` / `searchParams` は `Promise<...>`、fetch 既定は no-cache、`"use server"` の配置ルール（ファイル先頭 or 関数先頭、async 必須、Client 内禁止）、`remotePatterns` の書式（`{ protocol, hostname, pathname }`）
- 古い API 禁止: 仮想 DOM という用語、`pages/` ルーター、`getServerSideProps`、クラスコンポーネント、`forwardRef`
- Tailwind: v4（`@import "tailwindcss";`）準拠
- HTML / CSS: セマンティック、`box-sizing`、`prefers-color-scheme` ダーク対応
- TypeScript: `import type`、`interface` vs `type`、型ガード、判別共用体

## 出力（2000〜4000 字）
- 総評
- 技術的に誤り・古い（5〜12 件）: 引用 → 正しい API
- 不正確・誤解を生む（3〜8 件）
- コードが動かない / import 漏れ / 型が合わない（3〜6 件）
- 最新版との齟齬（2〜4 件）
- 総合評価: 技術正確性 / 最新版準拠度（各 10 段階）/ Top 5

辛口、引用と正答をペアで、絵文字禁止。
```

## エージェント 4: 通読 UX 視点

```
あなたは Web フロントエンド学習コンテンツの **レビュアー（通読 UX、実際に読んでつまずきを拾う）** です。コード変更は行わず、読んでレビューするだけ。

## 必読ファイル
1. /home/user/web-front-handson-ozaki25/CLAUDE.md
2. レビュー対象: <対象ファイル>

## 役割
読者になりきって対象を **冒頭から末尾まで通読** し、「読み進める間に詰まった / 理解に余計な手間がかかった」箇所を引用つきで挙げる。チェックリスト的に項目を埋めるのではなく、**読みながら違和感を覚えた瞬間を記録** するのが仕事。

## 観点（読みながら以下に引っかかったか）

1. **図表の伝達力**: 表に説明が薄く、何を言っているのか初読で掴めない箇所はあるか（数字だけ並んでいる / 列が抽象的すぎる / 例が無い）
2. **デモの差別化**: LiveDemo の見た目の違いが目で分かるか（色がグレー系で似ていて区別が付かない、複数概念をまとめたデモで何が要点か掴めない、など）
3. **用語の初出順序**: 説明なしで用語が登場していないか。とくに 1 文前で初めて出てきた語をすぐ別の文脈で使っていないか
4. **例と説明の対応ズレ**: 「○○すると Y になる」と書いてあるのに直後の例コードでは Y になっていない、説明文の主語と例の主語が違う、など
5. **演習の指示の明瞭さ**: 「次に / 続いて」のような順序語が指す対象がブレていないか、コピペするコードと前後の説明文が矛盾していないか
6. **段落の流れ**: 唐突な話題転換、前置きが長すぎて結論にたどり着くまでに迷子になる箇所
7. **読んだ直後に疑問が残る箇所**: 「これってどういう意味？」「ここで止まる」「この例だけだとピンと来ない」と感じた点
8. **構造的な迷子ポイント**: 見出し / 本文 / コード例 / デモの並び順が、読者の自然な思考順序と合っていない

チェックリストではなく **引用 → 違和感の言語化 → 改善案** を 1 件ずつ書く。

## 出力（2000〜4000 字）
- 総評: 通読時に何回詰まったか / 全体の読みやすさ評価
- Critical（3〜7 件）: 該当箇所引用 + 「自分が読み手として何で詰まったか」の説明 + 改善案
- High（5〜12 件）: 同
- Middle（3〜6 件）: 同
- 総合評価: 通読のスムーズさ（10 段階）/ 「初めて読む人がここでつまずく確率が高い」Top 5

辛口、引用ベース、絵文字禁止。「ルールに違反している」ではなく「読んで分からなかった」を主軸に書く。
```

## 統合レポート（4 エージェント完了後）

- 4 つの Top 5 を集約し、2 視点以上で収束した項目を **Converged Critical** として最優先
- 1 視点のみの指摘も列挙（特に通読 UX のみで挙がる「図表が伝わらない」「デモの差が見えない」は構造問題なので必ず一覧に残す）
- ユーザーに「このまま反映しますか？」と確認してから修正に入る

## 並列起動時の注意

- 5 エージェント同時以上だと API レート制限に当たりやすい。4 エージェントは経験上問題ないが、複数の章を同時にレビューしたい場合は章ごとに分割実行する
- タイムアウト対策として各エージェントの指示は簡潔に保つ（背景情報は必読ファイルへの参照で渡す）
