# カリキュラムレビュー（4 視点並列）

引数: `$ARGUMENTS`（例: `docs/lessons/lesson47/index.md` または `4 章`）

対象を 4 エージェント並列で辛口レビューし、指摘を統合してレポートする。過去のサイクルで実際に回った 3 視点に加え、図表・デモ・例の伝達力を見る「通読 UX」視点を 4 つ目として固定化したもの。

## 起動手順

1. レビュー対象のファイル / 範囲を `$ARGUMENTS` から決める（ファイル、ディレクトリ、章番号のいずれでも対応）
2. 以下 4 つのエージェントを **`run_in_background: true` で並列起動** する（`subagent_type: general-purpose`）
3. 4 エージェントすべて完了したら、収束した指摘を **Critical / High / Middle** で統合し、Top 5 を抽出してユーザーに報告

## エージェント 1: 初心者独学者視点

```
あなたは Web フロントエンド学習カリキュラムの **レビュアー（初心者独学者視点、完走可能性と流れ重視）** です。コード変更は行わず、読んでレビューするだけ。

## 必読ファイル
1. /home/user/web-front-handson-ozaki25/CLAUDE.md
2. レビュー対象: <対象ファイル>
3. 前後のレッスン（該当する場合）

## 役割
想定読者は「HTML / CSS / JS / React / Next.js をほぼ触ったことがない独学者」。完走可能性を最優先に評価。

## 観点
1. 脱落ポイント（レッスン番号 + 理由）
2. 前提の飛躍（前レッスンで未説明の概念を突然使っているか）
3. 期待出力の明確さ
4. コードの省略 `…(省略)…` の有無
5. 螺旋構造の機能（前後の接続）
6. 統合レッスンの敷居（前章までの知識で解けるか）
7. モチベーション維持
8. トーンの一貫性

## 出力（2000〜4000 字）
- 総評（300〜500 字）
- Critical（3〜7 件）: lessonXX の引用 + 改善案
- High（5〜12 件）: 同
- Middle（3〜6 件）
- 総合評価: 完走可能性（10 段階）/ 公開可能レベル判定 / 次サイクルで必ず潰すべき Top 5

辛口、具体的、絵文字禁止。
```

## エージェント 2: 技術正確性・Context7 照合視点

```
あなたは Web フロントエンド学習コンテンツの **レビュアー（技術正確性、Context7 によるドキュメント照合）** です。コード変更は行わず、読んでレビューするだけ。

## 必読ファイル
1. /home/user/web-front-handson-ozaki25/CLAUDE.md
2. レビュー対象: <対象ファイル>

## 作業手順（この順に必ず実行する）

### Step 1: ライブラリ API を列挙する
対象ファイルを読み、本文・コード例に登場する **すべてのライブラリ固有の記述** を抽出してリストアップする。
例: 関数名・フック名・設定オプション・ディレクティブ・型名・インポートパス・挙動の説明など。

### Step 2: Context7 でドキュメントを取得する
Step 1 で列挙した記述を対象ライブラリ別にグルーピングし、各ライブラリについて以下を実行する。
1. `resolve-library-id`（ライブラリ名 → Context7 ライブラリ ID）
2. `get-library-docs`（Step 1 の記述に関連するトピックをクエリに指定）
自分がすでに「知っている」と思っていても省略してはならない。

### Step 3: 照合する
Step 2 で取得したドキュメントと、本文の記述を 1 件ずつ照合する。
- 一致 → 問題なし
- 不一致・記述なし・バージョン差異あり → 指摘リストに追加（「本文の記述」→「ドキュメントに書かれている正しい内容」のペアで）

## 出力（2000〜4000 字）
- Step 1 で列挙したライブラリ API の一覧（照合済みであることの証跡）
- 照合結果: 問題なし件数 / 指摘件数
- 技術的に誤り・古い（件数上限なし）: 本文引用 → ドキュメントの正しい記述
- 不正確・誤解を生む記述（件数上限なし）
- コードが動かない / import 漏れ / 型が合わない
- 総合評価: 技術正確性 / 最新版準拠度（各 10 段階）/ Top 5

辛口、引用と正答をペアで、絵文字禁止。「自分の知識で確認した」は根拠にならない。Context7 で確認した事実だけを書く。
```

## エージェント 4: 通読 UX 視点

```
あなたは Web フロントエンド学習コンテンツの **レビュアー（通読 UX、実際に読んでつまずきを拾う）** です。コード変更は行わず、読んでレビューするだけ。

## 必読ファイル
1. /home/user/web-front-handson-ozaki25/CLAUDE.md
2. レビュー対象: <対象ファイル>

## 役割
読者になりきって対象を **冒頭から末尾まで通読** し、「読み進める間に詰まった / 理解に余計な手間がかかった」箇所を引用つきで挙げる。チェックリスト的に項目を埋めるのではなく、**読みながら違和感を覚えた瞬間を記録** するのが仕事。

## 観点（読みながら以下に引っかかったか）

1. **図表の伝達力**: 表に説明が薄く、何を言っているのか初読で掴めない箇所はあるか（数字だけ並んでいる / 列が抽象的すぎる / 例が無い）
2. **デモの差別化**: LiveDemo の見た目の違いが目で分かるか（色がグレー系で似ていて区別が付かない、複数概念をまとめたデモで何が要点か掴めない、など）
3. **用語の初出順序**: 説明なしで用語が登場していないか。とくに 1 文前で初めて出てきた語をすぐ別の文脈で使っていないか
4. **例と説明の対応ズレ**: 「○○すると Y になる」と書いてあるのに直後の例コードでは Y になっていない、説明文の主語と例の主語が違う、など
5. **演習の指示の明瞭さ**: 「次に / 続いて」のような順序語が指す対象がブレていないか、コピペするコードと前後の説明文が矛盾していないか
6. **段落の流れ**: 唐突な話題転換、前置きが長すぎて結論にたどり着くまでに迷子になる箇所
7. **読んだ直後に疑問が残る箇所**: 「これってどういう意味？」「ここで止まる」「この例だけだとピンと来ない」と感じた点
8. **構造的な迷子ポイント**: 見出し / 本文 / コード例 / デモの並び順が、読者の自然な思考順序と合っていない

チェックリストではなく **引用 → 違和感の言語化 → 改善案** を 1 件ずつ書く。

## 出力（2000〜4000 字）
- 総評: 通読時に何回詰まったか / 全体の読みやすさ評価
- Critical（3〜7 件）: 該当箇所引用 + 「自分が読み手として何で詰まったか」の説明 + 改善案
- High（5〜12 件）: 同
- Middle（3〜6 件）: 同
- 総合評価: 通読のスムーズさ（10 段階）/ 「初めて読む人がここでつまずく確率が高い」Top 5

辛口、引用ベース、絵文字禁止。「ルールに違反している」ではなく「読んで分からなかった」を主軸に書く。
```

## 統合レポート（4 エージェント完了後）

- 4 つの Top 5 を集約し、2 視点以上で収束した項目を **Converged Critical** として最優先
- 1 視点のみの指摘も列挙（特に通読 UX のみで挙がる「図表が伝わらない」「デモの差が見えない」は構造問題なので必ず一覧に残す）
- ユーザーに「このまま反映しますか？」と確認してから修正に入る

## 並列起動時の注意

- 5 エージェント同時以上だと API レート制限に当たりやすい。4 エージェントは経験上問題ないが、複数の章を同時にレビューしたい場合は章ごとに分割実行する
- タイムアウト対策として各エージェントの指示は簡潔に保つ（背景情報は必読ファイルへの参照で渡す）

## 全レッスンを横断レビューするときの進め方（三層分担）

上の 4 視点は「1 ファイル / 1 章を深く」見る縦のレビュー。全 138 レッスンを通す横断レビューでは、**役割を道具の得意分野で分ける**と無駄がない。

- **スクリプト（モデル不使用）**: 機械で判定できるものはモデルに渡さない。`textlint`、リンク実在、`config.mts` ↔ ディレクトリ ↔ H1 タイトル整合、hook の各パターン。
- **Fable（文章・日本語の質）**: 日本語の不自然さ・読みやすさ・トーン・構造破綻・AI 感語彙・スラッシュ列挙の**検出とリライトまで**。質が問われる場面はここに集中させる。技術的正誤には触らせない。
- **Opus（技術検証・運行）**: 技術正確性は**出典必須の手続き**として回す。記憶で判定しない → Context7（`resolve-library-id` → `get-library-docs`）→ 不通なら §6.7 フォールバック（公式 repo raw / `npm view` / `git ls-remote --tags`）→ 本文とドキュメントを 1 件ずつ照合し、各判定に出典を添える。出典を取れない項目は「未検証」に隔離し推測で正誤を書かない。あわせて章単位で `npm run docs:build` + hook ゲートを通してコミット。

技術検証で特に注意する **バージョン変動項目**: `fetch` のキャッシュ既定、`cookies()` / `params` の async 化、フレームワークやアクションのメジャー版、beta 版インストール（`next-auth@beta` 等）、ブラウザ対応バージョン。

### 他リポジトリのレビュー資産を参考にするときの無視リスト（引きずられ防止）

姉妹リポジトリ（`web-front-training-ozaki25` 等）のレビュースキルを参考にするのは有益だが、**前提が違うルールをそのまま適用すると誤指摘になる**。本コースでは次を欠陥として報告しない:

- 図が Mermaid でないこと（本コースは静的 SVG が正。§13）
- レッスンをまたぐ相互参照（螺旋構造が設計意図）
- 読者が初心者すぎる / レッスンが長すぎる（統合演習は 60〜120 分想定）
- 15 分読了・standalone 前提
